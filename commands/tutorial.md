cursor-power の使い方を体験的に学べるウォークスルーを実行する。

あなたは親エージェントとしてガイド役を務める。ダミータスクを使って cursor-power の一通りのフローをユーザーに体験させる。各ステップでは必ず「今何をしているか」を説明し、「次に何を入力すべきか」を具体的に提示し、ユーザーの入力を待ってから次に進むこと。

---

まず以下の導入メッセージを表示する:

```
cursor-power チュートリアルへようこそ！

このチュートリアルでは、ダミータスクを使って cursor-power の基本的なフローを一通り体験します。

  Step 1: /task-plan で仕様を対話的に決める
  Step 2: タスク登録 → 子エージェント起動を確認
  Step 3: /task-status で進捗を確認する
  Step 4: /task-check で子エージェントの質問に回答する
  Step 5: /task-review で PR をレビューする
  Step 6: マージまたはクローズする
  Step 7: /task-clean でクリーンアップする

それでは始めましょう！
```

---

## Step 1: /task-plan の対話フロー体験

ユーザーに以下を説明する:

```
━━━ Step 1/7: 仕様を対話的に決める (/task-plan) ━━━

最初のステップは /task-plan コマンドです。
実際のタスクでは、背景・目的・対象ファイル・ベースブランチをヒアリングして仕様を固めます。

今回はダミーの仕様を使って体験してみましょう。
以下をそのままコピー＆ペーストで入力してください:
```

以下の入力例を提示する:

```
背景: README.md にチュートリアルの案内がない
目的: README.md に「はじめての方へ」セクションを追加する
対象ファイル: README.md
ベースブランチ: main
```

ユーザーが上記（またはそれに類する内容）を入力したら、以下のように仕様をまとめて確認する:

```
仕様をまとめました:

## 背景
README.md にチュートリアルの案内がない

## 目的
README.md に「はじめての方へ」セクションを追加する

## 対象ファイル
README.md

## ベースブランチ
main

この内容で進めてよろしいですか？ → 「はい」と入力してください
```

ユーザーが承認したら Step 2 へ進む。

---

## Step 2: タスク登録 → 子エージェント起動を確認

ユーザーに以下を説明する:

```
━━━ Step 2/7: タスク登録と子エージェントの起動 ━━━

仕様が確定しました！
/task-plan では承認後に以下が自動的に実行されます:

  1. 仕様の保存（save-plan.mjs）
  2. ブランチ名の自動判定（type: docs, title: add-getting-started）
  3. タスクの登録（add-task.mjs）
  4. 子エージェントの起動（start-worker.mjs）

子エージェントは独立した worktree を作成し、そこでコードの実装 → コミット → プッシュ → PR 作成まで自動で行います。
あなたは待っているだけで OK です。

💡 なお、簡易的にタスクを登録したい場合は /task-add コマンドも使えます:
  /task-add README.mdに「はじめての方へ」セクションを追加する

/task-add でも --type と --title は自動判定されます。
並列実行の上限に達している場合、タスクは pending 状態で待機し、
枠が空いたら自動的に起動されます。
```

以下のようにシミュレーション結果を表示する:

```
（チュートリアルモード: 実際のタスク登録はスキップします）

実際に実行した場合、以下のような流れになります:

  ✓ プラン plan-abc123 を保存しました
  ✓ ブランチ名: docs-add-getting-started-xyz789
  ✓ タスク task-xyz789 を登録しました
  ✓ 子エージェントを起動しました

子エージェントが作業を開始すると、独立した worktree で以下を実行します:
  1. ブランチ docs-add-getting-started-xyz789 を作成
  2. README.md を編集
  3. git commit → git push
  4. gh pr create で PR を自動作成

次のステップで進捗の確認方法を学びましょう。
「次へ」と入力してください。
```

ユーザーが次へ進む意思を示したら Step 3 へ。

---

## Step 3: /task-status で進捗確認

ユーザーに以下を説明する:

```
━━━ Step 3/7: 進捗を確認する (/task-status) ━━━

子エージェントがバックグラウンドで作業している間、
/task-status コマンドでいつでも進捗を確認できます。

次のコマンドを入力してください:

  /task-status
```

ユーザーが入力したら、以下のようにシミュレーション結果を表示する:

```
（チュートリアルモード: シミュレーション結果を表示します）

タスク一覧:
┌──────────────┬────────────┬───────────────────────────────────┬──────────────┐
│ ID           │ ステータス │ 内容                              │ 更新         │
├──────────────┼────────────┼───────────────────────────────────┼──────────────┤
│ task-xyz789  │ pr_created │ README.mdに「はじめての方へ」セク… │ 1分前        │
└──────────────┴────────────┴───────────────────────────────────┴──────────────┘

ステータスの種類:
  • pending              — タスク登録済み、まだ未着手
  • running              — 子エージェントが作業中
  • blocked              — 子エージェントが質問して回答待ち（/task-check で確認）
  • acceptance_running   — 受け入れテスト実行中
  • fixing               — 受け入れテスト不合格、修正中
  • pr_created           — PR が作成済み、レビュー待ち
  • done                 — マージ完了
  • failed               — エラーで停止

💡 全タスクの一覧だけを見たい場合は /task-list も使えます。
💡 Web ブラウザでリアルタイム監視したい場合は /dashboard が便利です。

PR ができたようです！
ただし、その前に子エージェントからの質問への回答方法を学びましょう。
「次へ」と入力してください。
```

ユーザーが次へ進む意思を示したら Step 4 へ。

---

## Step 4: /task-check で子エージェントの質問に回答

ユーザーに以下を説明する:

```
━━━ Step 4/7: 子エージェントの質問に回答する (/task-check) ━━━

子エージェントが判断に迷ったとき、質問ファイルを書いて回答を待ちます。
ステータスが blocked になっていたら /task-check で質問を確認・回答できます。

次のコマンドを入力してください:

  /task-check
```

ユーザーが入力したら、以下のようにシミュレーション結果を表示する:

```
（チュートリアルモード: シミュレーション結果を表示します）

task-xyz789 から質問:
  「"はじめての方へ" セクションは README の冒頭に置きますか？
    それとも "基本的な使い方" セクションの前に置きますか？」

→ 何か回答を入力してみてください（例: 「基本的な使い方の前に置いてください」）
```

ユーザーが回答を入力したら:

```
（チュートリアルモード: 回答送信をシミュレーションします）

実際の動作では:
  1. 質問ファイルに回答が書き込まれる（check-questions.mjs）
  2. 子エージェントに回答が中継される（send-answer.mjs）
  3. 子エージェントが回答を受けて作業を再開する

task-xyz789 に回答を送信しました。
次のステップでレビュー方法を学びましょう。
「次へ」と入力してください。
```

ユーザーが次へ進む意思を示したら Step 5 へ。

## Step 5: /task-review でレビュー体験

ユーザーに以下を説明する:

```
━━━ Step 5/7: PR をレビューする (/task-review) ━━━

子エージェントが PR を作成したら /task-review でレビューできます。
タスク概要・安全度スコア・変更ファイルの一覧を確認し、
diff を開き、必要なら修正指示を送れます。

次のコマンドを入力してください:

  /task-review
```

ユーザーが入力したら、以下のようにシミュレーション結果を表示する:

```
（チュートリアルモード: シミュレーション結果を表示します）

タスク task-xyz789 のレビュー (PR: https://github.com/user/repo/pull/99)

概要: README.md に「はじめての方へ」セクションを追加（ベース: main）
安全度: 影響の小ささ 5/5, 発生しにくさ 5/5

変更ファイル:
  1. README.md (+15 / -0) — 「はじめての方へ」セクションを追加

見たいファイルを選んでください（番号またはファイル名）。
→ 「1」と入力してください
```

ユーザーが「1」（または README.md）を入力したら:

```
（チュートリアルモード: 通常はここでエディタに diff が表示されます）

diff の表示イメージ:

  + ## はじめての方へ
  +
  + cursor-power を初めて使う方は `/tutorial` コマンドを実行すると、
  + 基本的なフローを体験できます。
  + ...

diff を見て問題がなければ「OK」、修正してほしい点があれば修正指示を入力します。
例: 「セクションのタイトルを "クイックスタート" に変更して」

実際に試してみましょう。何か修正指示を入力するか、「OK」と入力してください。
```

ユーザーが修正指示を入力した場合:

```
（チュートリアルモード: 修正指示の送信をシミュレーションします）

実際の動作では:
  1. 修正指示が子エージェントに送信される（send-answer.mjs）
  2. 子エージェントが指示に従いコードを修正
  3. 新しいコミットが PR に追加される
  4. 再度 /task-review で確認できる

修正指示を送信しました。次のステップへ進みましょう。
「次へ」と入力してください。
```

ユーザーが「OK」と入力した場合:

```
レビュー完了です！次のステップでマージ方法を学びましょう。
「次へ」と入力してください。
```

ユーザーが次へ進む意思を示したら Step 6 へ。

## Step 6: マージまたはクローズ

ユーザーに以下を説明する:

```
━━━ Step 6/7: PR をマージまたはクローズする ━━━

レビューが完了したら、PR をマージまたはクローズします。

マージする場合:
  GitHub 上で「Merge pull request」をクリックするか、
  ターミナルで以下を実行します:

    gh pr merge <PR番号> --squash

クローズする場合（変更を取り消したい場合）:

    gh pr close <PR番号>

これらは cursor-power のコマンドではなく、通常の GitHub 操作です。
マージ後のクリーンアップは次のステップで行います。

「次へ」と入力してください。
```

ユーザーが次へ進む意思を示したら Step 7 へ。

---

## Step 7: /task-clean でクリーンアップ

ユーザーに以下を説明する:

```
━━━ Step 7/7: クリーンアップする (/task-clean) ━━━

PR がマージまたはクローズされたら /task-clean でクリーンアップできます。
マージ済み・クローズ済みの worktree、ブランチ、質問ファイル、ログがまとめて削除されます。
タスクに紐づく issue がある場合は、同時に issues.json からも削除されます。

次のコマンドを入力してください:

  /task-clean
```

ユーザーが入力したら、以下のようにシミュレーション結果を表示する:

```
（チュートリアルモード: シミュレーション結果を表示します）

クリーンアップ結果:
  ✓ task-xyz789
    - worktree 削除: ~/.cursor/worktrees/repo/docs-add-getting-started-xyz789
    - ブランチ削除: docs-add-getting-started-xyz789
    - 質問ファイル削除: ~/.cursor-power/questions/task-xyz789.json
    - ログ削除: ~/.cursor-power/logs/task-xyz789.log
    - タスクステータス: done に更新

未マージ・未クローズのタスクはスキップされます。安心して実行できます。
```

最後に以下の完了メッセージを表示する:

```
━━━ チュートリアル完了！ ━━━

お疲れさまでした！cursor-power の基本フローを一通り体験しました。

おさらい — よく使うコマンド:
  /task-plan    … 仕様を対話的に決めてタスクを起動（推奨）
  /task-add     … 簡易的にタスクを登録・起動
  /task-list    … 全タスクの一覧を表示
  /task-status  … 子エージェントの進捗を確認
  /task-check   … 子エージェントからの質問に回答
  /task-review  … PR のレビューと修正指示
  /task-clean   … マージ済みタスクのクリーンアップ
  /task-config  … cursor-power の設定変更

おさらい — その他のコマンド:
  /issue-add    … アイデアや改善点を issue として記録
  /issue-list   … 登録済み issue の一覧を表示
  /task-promote … issue を対話で仕様を詰めてタスクに昇格
  /dashboard    … タスク状態を監視する Web ダッシュボードを起動

💡 受け入れテスト: /task-plan や /task-add で --acceptance を付けると、
   PR 作成前に自動で受け入れテストを実行できます。

さっそく /task-plan で最初のタスクを始めてみましょう！
```
